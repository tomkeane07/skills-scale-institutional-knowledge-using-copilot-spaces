# OctoAcme Project Management Docs

Welcome to the OctoAcme project management documentation. This folder is the single source of truth for how OctoAcme plans, executes, and continuously improves software delivery.

## Overview

OctoAcme runs projects according to a **customer-first, goal-oriented** delivery model. Work is broken into small, testable increments so teams can learn quickly, reduce risk, and adapt to feedback rather than waiting for a large-batch release. Every project has a named Project Manager and Product Manager to ensure **clear ownership** at both the delivery and outcome levels.

The delivery lifecycle moves through five stages: **initiation** (problem statement, stakeholders, high-level timeline), **planning** (scope, backlog, milestones, dependencies, Definition of Done), **execution** (build, test, review, iterate), **release** (deploy, verify, announce), and **retrospective / continuous improvement** (capture learnings, convert them into tracked action items). Teams maintain a regular cadence of standups, weekly delivery syncs, and end-of-sprint demos to keep work visible and blockers surfaced early.

**Communication and escalation** follow a defined path: blockers are first triaged by the team, escalated to the PM when unresolved, and raised to sponsor level when business impact is significant. Stakeholders receive weekly status updates and milestone-based reports with a single source of truth for project status.

**Quality assurance** is built into every stage, not added at the end. Acceptance criteria must be met, CI and security scans must pass, release notes must be prepared, and rollback plans must exist before production deployment. Testing expectations include unit, integration, smoke, security, and manual QA as appropriate.
